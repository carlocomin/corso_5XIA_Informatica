Livello logico vs livello fisico

Obiettivo

  • Distinguere in modo rigoroso schema logico e schema fisico.
  • Definire l'indipendenza dei dati (logica e fisica) e le sue conseguenze pratiche.
  • Collegare il modello relazionale a scelte implementative: indici, file organization, piani di esecuzione.

Architettura ANSI/SPARC (3 livelli)

  • Esterno: viste e interfacce utente/applicazione (sottoinsiemi/riformulazioni dello schema concettuale).
  • Concettuale (logico): schemi di relazione, domini, vincoli (PK, FK, ecc.).
  • Interno (fisico): strutture di memorizzazione, pagine, indici, partizioni, compressione.

Il modello relazionale vive al livello concettuale; le query sono dichiarative e indipendenti dal livello interno.

Indipendenza dei dati

  • Indipendenza fisica: cambi al livello interno (es. aggiungere un indice, cambiare organizzazione a pagine) non richiedono modifiche a schemi e query logiche.
  • Indipendenza logica: cambi compatibili al livello concettuale (es. aggiungere un attributo o scindere una tabella) non rompono le interfacce esterne; si preservano con viste e regole di compatibilità all’indietro.

Rinominare attributi senza mantenere viste o compatibilità rompe l’indipendenza logica.

Logico: cosa definisce

  • Schemi di relazione: intestazioni, domini, vincoli (integrità di dominio, entità, referenziale, regole aggiuntive).
  • Semantica delle operazioni (algebra relazionale, calcolo relazionale): nessun riferimento a strutture fisiche.
  • Identità: chiavi primarie/candidate e loro significato.

Fisico: cosa riguarda

  • Organizzazione dei file: heap, clustered, append-only.
  • Indici: B-Tree/B+Tree, Hash, bitmap; indici clustered vs nonclustered, indici di copertura.
  • Layout: pagine, fillfactor, compressione, partizionamento (range/hash/list), replica.
  • Identificatori fisici: RID/ctid/rowid non fanno parte del modello logico.

Una chiave è logica; un indice è fisico. Molti DB implementano PK con un indice UNIQUE, ma concettualmente sono distinti.

Ottimizzazione delle query

  • L’utente scrive SQL dichiarativo; l’ottimizzatore genera un piano fisico equivalente (scan, index scan, join, sort, ecc.).
  • Il piano dipende da statistiche e indici disponibili, non dalla forma sintattica della query (entro certi limiti).
-- Stessa logica, fisica diversa a seconda degli indici disponibili
-- Caso A: nessun indice su Iscrizioni(Matr)
EXPLAIN SELECT *
FROM Iscrizioni i JOIN Studenti s ON i.Matr = s.Matr
WHERE s.Matr = 101;
-- Piano tipico: Seq Scan Iscrizioni → Hash Join → Filter

-- Caso B: indice su Iscrizioni(Matr) e PK indicizzata su Studenti(Matr)
EXPLAIN SELECT *
FROM Iscrizioni i JOIN Studenti s ON i.Matr = s.Matr
WHERE s.Matr = 101;
-- Piano tipico: Index Seek/Scan → Nested Loop → molto più veloce

Esempi di scelte fisiche

Linee guida sintetiche
  • Indici B-Tree per ricerche per uguaglianza e per intervallo; Hash per uguaglianze pure (dove supportati); bitmap per colonne a bassa cardinalità (DW).
  • Partizionare tabelle molto grandi per migliorare manutenzione e pruning delle scansioni.
  • Valutare indici di copertura per query di lettura frequenti; evitare over-indexing che rallenta le scritture.
  • Preferire PK immutabili; ON UPDATE CASCADE diventa raro e i piani restano stabili.

Compatibilità e migrazioni

  • Compatibilità logica: usare view per mantenere API stabili quando lo schema concettuale evolve.
  • Compatibilità fisica: cambiare indici o partizioni non dovrebbe toccare le query; valutare re-build/analisi statistiche.
Domanda-guida: perché una chiave primaria non coincide con un indice?
  • Natura: la PK è un vincolo logico (identità e unicità); l’indice è una struttura fisica per accelerare accessi.
  • Implementazione: un DB spesso usa un indice UNIQUE per far rispettare la PK, ma può farlo con altre tecniche; l’esistenza dell’indice è una scelta di implementazione.
  • Conseguenze: puoi cambiare/ricreare indici senza toccare la semantica dello schema; la PK invece fa parte dello schema e delle sue dipendenze (FK).
-- Vincolo logico vs struttura fisica
CREATE TABLE Studenti (
  Matr   INTEGER NOT NULL,
  Nome   VARCHAR(50) NOT NULL,
  Email  VARCHAR(100) NOT NULL,
  CONSTRAINT PK_Studenti PRIMARY KEY (Matr)
);

-- Unicità logica su Email (vincolo), realizzata tipicamente con indice UNIQUE
ALTER TABLE Studenti ADD CONSTRAINT UQ_Studenti_Email UNIQUE (Email);

-- Struttura fisica aggiuntiva per performance (nessun effetto sulla semantica)
CREATE INDEX IX_Iscrizioni_Matr ON Iscrizioni(Matr);
