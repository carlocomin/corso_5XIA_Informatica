Dependency preservation (preservazione delle dipendenze)

Definizione (idea pratica)

Data una decomposizione di R in (R1, R2, ..., Rn), diciamo che essa preserva le dipendenze se:

ogni vincolo funzionale di interesse si può controllare imponendo DF solo sulle singole relazioni Ri, senza ricostruire R con un join.

Se una DF “vive” su attributi distribuiti su più tabelle, potrebbe non essere verificabile localmente.

Intuizione: è la differenza tra “posso controllare la regola guardando una tabella” e “per controllarla devo unire tabelle e guardare il risultato globale”.

Nota critica: BCNF vs preservazione

L’algoritmo di decomposizione in BCNF garantisce in genere:

  • join senza perdita (lossless)
  • ma non sempre la preservazione delle dipendenze

Questo è uno dei motivi per cui, in alcuni casi, si preferisce una decomposizione in 3NF (lossless e dependency-preserving) invece di spingersi fino a BCNF.

Controesempio classico (mostra il problema)

Schema:

R(A, B, C)

DF:

F = { AB → C,  C → B }

La DF C → B viola BCNF (in genere C non è superchiave), quindi decomponiamo su C → B:

R1(C, B)
R2(A, C)

Questa decomposizione resta lossless: R1 ∩ R2 = {C} e in R1 vale C → B, quindi C è chiave di R1 e fa da collante.

Dopo la decomposizione, la DF AB → C non “sta” più in una singola tabella (A è in R2, B è in R1).

Dimostrazione concreta: vincoli locali ok, vincolo globale no

Considera le seguenti istanze che rispettano i vincoli locali:

r1(C, B) = { (c1, b1), (c2, b1) }
  - qui vale C → B

r2(A, C) = { (a1, c1), (a1, c2) }

Il join naturale produce:

r1 ⋈ r2 = { (a1, b1, c1), (a1, b1, c2) }

Qui AB → C è violata: la coppia (a1, b1) determina due valori diversi di C (c1 e c2).

Conclusione: decomposizione lossless, ma NON dependency-preserving.

Conseguenza pratica (scelta progettuale)

  • BCNF: più “pulita” contro ridondanze da DF, ma può perdere la verificabilità locale di alcuni vincoli.
  • 3NF: spesso permette decomposizioni lossless e dependency-preserving, accettando qualche ridondanza residua in casi particolari.

La scelta dipende da cosa è più importante nel contesto: eliminare ridondanza al massimo oppure poter imporre vincoli senza join.

Micro-verifica finale

Domanda: qual è la differenza tra “lossless join” e “dependency preservation”?

  • Lossless join: dal join delle tabelle decomposte posso ricostruire l’informazione originale senza “inventare” associazioni.
  • Dependency preservation: posso controllare tutte le DF originali imponendo vincoli solo sulle singole tabelle, senza join.

Sono proprietà diverse: una può valere senza l’altra.
